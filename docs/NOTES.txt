## Module Build Order (Dependency Chain)

**IMPORTANT:** Developer 1 must complete M1 first — all other modules depend on it.
Then M2 (Auth) must complete before Developers 2/3/4 can start.

```
M1 (Foundation) ←── ALL modules depend on this
  └── M2 (Auth) ←── ALL authenticated modules depend on this
        ├── M3 (Profile)
        ├── M4 (Dashboard) ←── needs M3, M5
        ├── M5 (Trips) ←── Developers 2/3/4 can start here
        │     ├── M6 (Itinerary) ←── needs M5
        │     ├── M7 (Search) ←── needs M5
        │     ├── M8 (Budget) ←── needs M5
        │     ├── M9 (Packing) ←── needs M5
        │     ├── M10 (Notes) ←── needs M5
        │     ├── M11 (Collaboration) ←── needs M5, M14
        │     ├── M12 (Sharing) ←── needs M5
        │     └── M13 (AI Services) ←── needs M5, M6, M8, M9
        ├── M14 (Notifications)
        ├── M15 (Community) ←── Phase 2, needs M5, M12
        └── M16 (Admin) ←── Phase 2, needs M15
```

**Parallel execution strategy:**
1. Developer 1 builds M1 → M2 (sequential, ~1 hour)
2. Once M1+M2 are pushed, ALL developers can start their modules in parallel
3. Developer 1 continues with M3, M4
4. Developers 2/3/4 start their modules simultaneously

## Database: Supabase-Only (Simplified)
- All modules use Supabase PostgreSQL directly via Prisma
- NO local PostgreSQL in M1-M16
- Local PostgreSQL failover deferred to M17 (post-launch)
- This significantly simplifies initial development

## Seed Data Required
- 50 popular destinations (cities worldwide with metadata)
- 200 activities (across categories, linked to destinations)
- Create seed script in `prisma/seed.ts`

## Key Architecture Decisions
1. **Supabase-only database** (local PG failover deferred to M17)
2. Next.js API Routes (not separate NestJS backend) — reduces complexity
3. Supabase Auth instead of Clerk — zero additional cost
4. Google Gemini instead of OpenAI — free tier availability
5. Leaflet instead of Mapbox — no API key, truly free
6. PostHog instead of custom analytics — feature flags included
