# M17 — Local PostgreSQL Fallback Setup

**Priority:** Deferred (Post-MVP) | **Phase:** Post-Launch | **Complexity:** High

## Description
Add a local PostgreSQL database as a failover/redundancy layer alongside Supabase. When Supabase has connectivity issues, the app automatically switches to the local PostgreSQL database. Includes health monitoring, automatic failover, sync queue, and conflict resolution.

## Acceptance Criteria
- [ ] Local PostgreSQL 16 setup and schema sync with Supabase
- [ ] Prisma multi-database connection configuration
- [ ] DatabaseService class with health monitoring (30s heartbeat to Supabase)
- [ ] Automatic failover: Supabase down → route queries to local PG
- [ ] Write-ahead log for offline mutations (stored in IndexedDB for PWA)
- [ ] Sync queue: replay pending changes when Supabase reconnects
- [ ] Conflict resolution: Last-Write-Wins (LWW) using updated_at timestamps
- [ ] Schema migration tooling to keep both databases in sync
- [ ] Health status indicator in UI (connected/offline mode)
- [ ] All existing services updated to use DatabaseService instead of direct Prisma calls

## Database Touchpoints
- ALL tables (mirror Supabase schema to local PostgreSQL)
- New: `sync_queue` table (local only) for pending mutations

## API Touchpoints
- GET /api/v1/health (database connection status)
- All existing API routes updated to use DatabaseService

## Dependencies
- M1 (Foundation) — must be modified to add Prisma dual-connection
- All modules (M2-M16) — services must route through DatabaseService
